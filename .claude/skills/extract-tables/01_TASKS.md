# Task List: /extract-tables Phase 2 — Native Rust+Python Pipeline

**Created**: 2026-03-05
**Goal**: Replace Camelot Python backend with native Rust+compiled-Python, adding table titles, cross-page merging, normalized bboxes, and concurrent extraction support.

## Context

Camelot is a memory hog and very slow. We're replacing its Python backend with:
- **pdf_oxide** for text extraction + page rendering (replaces pdfminer + ghostscript/pypdfium2)
- **Rust image_proc** via PyO3 for lattice parser hotspots (replaces OpenCV Python)
- **Compiled Python** (mypyc) for parser algorithms
- **polars** for DataFrame output (replaces pandas)

The skill must be self-contained with: strategy routing (Shadow-LEGO), VLM title inference,
cross-page table merging, PyMuPDF-compatible bboxes (top-left origin), and shadow logging.

**Key design**: PDF text extraction and table detection run concurrently, then stitch results.

### CRITICAL: pdf_oxide coordinate system

pdf_oxide spans use `bbox = (x, y, width, height)` in **PDF bottom-left origin** (y=0 is bottom
of page, y increases upward). This is NOT top-left origin.

Conversion to top-left `(x0, y0_top, x1, y1_bottom)`:
```python
x0 = span.bbox[0]
y0_top = page_height - span.bbox[1] - span.bbox[3]  # flip y, subtract height
x1 = span.bbox[0] + span.bbox[2]
y1_bottom = page_height - span.bbox[1]
```

### CRITICAL: Do NOT use pdf_oxide.extract_tables()

pdf_oxide v0.3.14 exposes `PdfDocument.extract_tables()` but it returns 0 tables on all
Camelot test fixtures with all strategy combinations. It is not functional for our use cases.
Our parser pipeline uses `extract_spans()` and `render_page()` only.

## Notes

### Coordinate Convention (NON-NEGOTIABLE)
All external-facing bboxes use **top-left origin**: `(x0, y0_top, x1, y1_bottom)` where y=0 is the
top of the page. This matches PyMuPDF, PIL, and browser conventions.

**pdf_oxide returns `(x, y, w, h)` in bottom-left origin.** Conversion happens ONCE in
`pdf_bridge.py` and is never exposed to consumers. Every downstream module receives top-left bboxes.

### Concurrency Model
Pages are extracted independently and can run concurrently via `ProcessPoolExecutor`.
Cross-page merging happens AFTER all pages complete. This enables /extract-pdf to fire
table extraction alongside text extraction and stitch results.

### Phase 1 → Phase 2 Migration
Task 12 removes the Camelot delegation path. Until then, both backends can coexist.
The `_BACKEND` flag in `extract_tables.py` switches between them.

### imageproc Rust Crate (VERIFIED)
Using `imageproc 0.26.1` + `image 0.25.x` (pure Rust, no C++ dependency):
- `imageproc::contrast::adaptive_threshold(image, block_radius, delta)` — confirmed
- `imageproc::morphology::{erode, dilate, open, close}` with `Norm` or custom `Mask` — confirmed
- `imageproc::contours::find_contours()` — confirmed
- `imageproc::hough::detect_lines()` — confirmed (backup for line detection)
- No opencv-rust or C++ OpenCV needed.

### VLM Graceful Degradation
VLM title inference is OPTIONAL. If /scillm is unavailable (rate limit, no API key, network error),
the skill returns `ai_title=None` and continues. Tables without titles get `title=None`.
The skill NEVER fails due to VLM unavailability.

### pdf_oxide.extract_tables() — DO NOT USE
pdf_oxide v0.3.14 exposes `PdfDocument.extract_tables()` but it returns 0 tables on all Camelot
test fixtures. Our pipeline uses `extract_spans()` + `render_page()` to build tables from scratch.
