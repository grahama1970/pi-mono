What happens when a mission is paused?
When a mission is paused, three things happen simultaneously:

1. State machine M transitions running → paused
2. Event mission_paused fires via droid.session_notification to all connected clients
3. Each active worker receives worker_paused

The daemon holds the mission state at paused until daemon.resume_automation is called, which emits mission_resumed and restarts the workers.

The orchestrator_turn state suggests there's an intermediate state where the orchestrator (not a worker) has control — likely for reviewing worker output before continuing.
How does MCP authentication work?
The MCP auth flow is a 4-step OAuth-style handshake:

1. Client calls daemon.add_mcp_server with server URL
2. Server responds with mcp_auth_required event (needs user action)
3. User completes auth in browser, then calls droid.submit_mcp_auth_code with the code
4. mcp_auth_completed event fires, server status changes via mcp_status_changed

Cancellation: daemon.cancel_mcp_auth aborts mid-flow. daemon.clear_mcp_auth revokes existing credentials.

Both droid.* and daemon.* namespaces expose identical MCP methods — the client can manage MCP servers directly in CLI mode without the daemon.
What's the speculation mode?
Schema by0 has two model fields: modelId and specModeModelId, each with a separate reasoningEffort. This is speculative execution:

A fast model (specModeModelId, likely Haiku) generates a draft response while the user is still reading. The primary model (modelId, likely Sonnet/Opus) then validates or corrects it. If the speculation was correct, the user sees instant response. If not, the primary model's output replaces it.

The autonomyLevel field likely gates whether speculation runs automatically or waits for user confirmation.

This is droid — a 153 MB ELF binary built as a Bun Single-Executable Application targeting x86_64 Linux. It is an AI-powered coding agent with a client-daemon architecture: users interact through a CLI or GUI client that communicates with a long-running daemon process over JSON-RPC via WebSocket.

The investigation extracted 82 RPC methods across three namespaces, 59 event types grouped into 15 semantic families, 6 state machine enums (of which 3 contain noise from minification), 9 Zod validation schemas with significant field overlap, and 303 AST symbols from the embedded JavaScript bundle. Extraction was performed using a three-tier approach: T0 deterministic regex on string literals, T1 AST-based heuristic analysis, and T2 LLM-guided interpretation of ambiguous structures.

The most architecturally significant finding is the mission orchestration subsystem: droid can decompose a user goal into features, assign each to a parallel worker, pause/resume individual workers, and report progress through a heartbeat mechanism. This goes well beyond a simple chat agent — it is a task execution engine with job scheduling, terminal multiplexing, and git integration.

Section 01

What is this binary?

T0 DETERMINISTIC ELF headers + string extraction

Let's start with what we can say with absolute certainty. The ELF headers tell us this is an x86_64 dynamically-linked binary, 153 MB, not stripped. It links against standard glibc (libc.so.6, libpthread.so.0, libdl.so.2, libm.so.6) plus the dynamic linker itself. No exotic shared libraries — all the interesting logic is in the embedded JavaScript.

The Bun SEA (Single-Executable Application) format packs a complete JavaScript runtime plus bundled application code into a single ELF binary. We carved 16 bundled JS asset files from the $bunfs/root/ virtual filesystem embedded in the binary's data section. These assets contain the minified application code, and they're our primary analysis target.

Section 02

How does it communicate?

T0 T2 82 RPC methods · 3 namespaces

Droid speaks JSON-RPC over WebSocket. Every method we extracted follows the namespace.method_name pattern, and there are exactly three namespaces. Understanding this split is the key to understanding the architecture.

The droid.* namespace (25 methods) is the interactive session API. This is what the user touches: sending messages, granting permissions, managing tool confirmations. Think of it as the thin client — it knows about the conversation but delegates all heavy work to the daemon.

The daemon.* namespace (55 methods) is the persistence and execution layer. It owns terminals (PTY multiplexing), automations (scheduled tasks with cron expressions), git operations (commit, push, PR creation, semantic diffs), session lifecycle, and filesystem access. When you ask droid to "commit my changes," the client relays that to daemon.git_commit which actually runs the git operations.

The tunnel.* namespace (2 methods) is minimal: just init and ready. This is the handshake for the computer CLI command — a relay protocol for remote machine access.

The most telling detail in the RPC surface is the 14 MCP methods duplicated across both namespaces. Both droid.* and daemon.* expose identical methods for adding, removing, toggling, and authenticating MCP servers. This isn't an accident — it means the CLI client can manage MCP servers directly (without the daemon running), while the GUI routes everything through the daemon. Two paths to the same capability, unified by namespace mirroring.

The correlation key is sessionId. It appears in 23 methods across both namespaces, linking every client request to its daemon-side state. When you see sessionId in a schema, that field is the thread tying together the conversation, the terminals it spawned, the automations it triggered, and the git operations it performed.

⚠ What Could Go Wrong

The daemon.start_code_server method suggests the daemon can start a code-server (VS Code in the browser) instance. If this binds to 0.0.0.0 instead of 127.0.0.1, any machine on the network gets a code editor with the daemon's permissions. Combined with daemon.validate_working_directory and daemon.list_files, an unauthenticated or weakly-authenticated daemon could expose the entire filesystem.

The tunnel system (tunnel.init / tunnel.ready + computer CLI) is a remote execution mechanism. How does it authenticate? Does the relay server validate both ends? Without more context, this is the highest-risk surface in the binary.

Section 03

What are its internal states?

T1 AST T2 LLM 6 enum groups · 3 contain minifier noise

We extracted 6 enum groups from the minified JavaScript using the Variable.State="value" pattern. Here's the critical insight: three of these enums are "polluted" by the minifier, which packs multiple unrelated enums into the same variable to save bytes. The LLM tier was essential for separating signal from noise.

Mission Lifecycle (variable M, 12 states) — This is the heart of droid's orchestration engine. A mission starts when the user submits a goal. The system decomposes it into features, initializes workers, and runs them in parallel. Any worker can be paused independently for human review, and there's a special orchestrator_turn state where the coordinator (not a worker) takes control to evaluate progress and decide what to do next.

awaiting_input → initializing → running → paused → orchestrator_turn → completed
Minifier noise (same variable): team team_annual payg enterprise fast custom
▼ Noise Classification Rationale T2 LLM

The second group (team, team_annual, payg, enterprise, fast, custom) are pricing tier labels, not mission states. Evidence: (1) they have no corresponding events (no mission_state_changed ever carries these values), (2) the naming convention is completely different from the lifecycle states, (3) payg = "pay as you go" and team_annual are clearly subscription types. The minifier packed two unrelated enums into variable M.

Mission lifecycle state machine · T1 extraction + T2 interpretation

Connection + Conversation (variable B, 24 states) — Three completely different enums crammed into one variable by the minifier. The first is the WebSocket connection state machine: connecting → connected → disconnected → failed → disabled. Standard reconnection logic.

The second is the conversation turn state machine, and it's the one that drives the UI moment-to-moment. When the agent is thinking, the state is streaming_assistant_message. When it wants to run a tool, it enters waiting_for_tool_confirmation — the client shows a permission dialog. After the user approves, it moves to executing_tool. Between turns, compacting_conversation handles context window management. This is the pulse of the entire user experience.

WS: connecting → connected → disconnected → failed → disabled
Turn: idle → streaming_assistant_message → waiting_for_tool_confirmation → executing_tool → compacting_conversation
CSS noise: white browser swatch solid dashed dialog mermaid openai

LLM Provider (variable F, 10 values) — Not a state machine at all. This is a selector enum: anthropic, bedrock, vertex_anthropic, bedrock_anthropic, openai, azure_openai, google, fireworks, baseten, none. Anthropic is the primary provider. The Bedrock and Vertex variants indicate enterprise deployments that route through AWS or GCP rather than hitting Anthropic directly. fireworks and baseten are inference platforms for open-weight models.

Permission Resolution (variable T, 11 values) — The tool permission escalation ladder. When droid wants to run a tool: proceed_once (allow this one call), proceed_always (always allow this tool), proceed_auto_run (auto-approve below a risk level), with granular levels low/medium/high. The user can also proceed_edit (modify the tool call before running it), cancel (reject this call), or stop (kill the session). This maps directly to the droid.request_permission RPC method and the permission_resolved event.

⚠ What Could Go Wrong

The proceed_auto_run_high permission level is dangerous by design. Once a user sets autonomy to "high," the agent can execute any tool — including terminal commands, file writes, and git operations — without asking. If the agent is tricked via prompt injection in a file it reads, that injection inherits the auto-run permissions. The proceed_edit escape hatch suggests the developers know this is a concern.

The event system tells us more about the architecture than the RPC methods do, because events reveal what the system thinks is important enough to broadcast. It means missions are expected to be long-running — minutes to hours — and the system needs a liveness signal to distinguish "still working" from "hung." The heartbeat, combined with mission_progress_entry, tells us the GUI likely has a progress timeline or dashboard for watching missions unfold in real time.

droid_working_state_changed connection_changed assistant_message_delta assistant_message_completed user_message error

The worker model is two-level. A mission decomposes into features (tracked by mission_features_changed). Each feature is assigned to a worker. The lifecycle reads: mission_accepted → mission_run_started → N × (worker_started → worker_selected_feature → worker_completed | worker_failed) → mission_state_changed(completed). Workers can be paused independently without halting the mission. This is a proper job scheduler, not just a chat loop.

The thinking events are also architecturally significant. thinking_text_delta streams Claude's chain-of-thought reasoning to the client in real time. redacted_thinking is specifically for extended thinking output that gets redacted — the event exists so the UI can show "thinking..." without revealing the content. This tells us the binary has first-class support for Anthropic's extended thinking feature.

⚠ What Could Go Wrong

The session_token_usage_changed event broadcasts token counts to connected clients. If the WebSocket connection isn't authenticated per-session, one client could subscribe to another session's events, leaking the content of private conversations via token usage patterns (timing attacks) or direct event content.

Section 06

How does authentication work?

T2 LLM Deep-dive from interactive investigation

"Walk me through how a client authenticates with the daemon. What's the exact sequence?"

Authentication happens at daemon connection time, before any session work begins. Here's the sequence we can reconstruct from the schema and RPC evidence:

Step 1: WebSocket connect. The client opens a WebSocket to the daemon. Connection state enters connecting (state machine B). No auth yet — this is just transport.

Step 2: daemon.authenticate call. The client sends schema ax0 as the payload. It includes either a token (OAuth-style, from a browser login flow) or an apiKey (for programmatic access). The method field discriminates which auth path to take. Alongside the credential, the client sends metadata: platform, droidCLIVersion, isDroidCLIInPath, homedir. This metadata isn't for auth — it's telemetry and compatibility checking.

Step 3: Org resolution. The userId and orgId fields in the auth schema suggest multi-tenant support. After authentication, the daemon resolves which organization the user belongs to. This determines quota limits (the pricing tiers from state machine M: team, payg, enterprise).

Step 4: Connection established. State transitions to connected. The client receives a connectionId for this session's WebSocket. From here, the client can call any droid.* method.

"What about MCP server authentication? That's a separate flow, right?"

Completely separate. MCP (Model Context Protocol) servers are external extensions — third-party services that provide additional tools to the agent. Their auth is a 4-step OAuth-style handshake initiated by the user, not the daemon's own authentication:

1. Client calls daemon.add_mcp_server with the server URL.
2. Daemon contacts the MCP server. If it requires auth, mcp_auth_required event fires to the client.
3. The user completes authentication in a browser (OAuth redirect), then calls droid.submit_mcp_auth_code with the authorization code.
4. Daemon exchanges the code for a token. mcp_auth_completed fires. The server's status changes via mcp_status_changed.

Cancellation at any point via droid.cancel_mcp_auth. Credential revocation via droid.clear_mcp_auth. Both namespaces expose these methods because CLI users bypass the daemon for MCP management.

⚠ What Could Go Wrong

The daemon authentication schema accepts apiKey as an alternative to token. API keys are long-lived, don't rotate automatically, and are stored somewhere on disk. If the daemon binds to 0.0.0.0 (which start_code_server suggests is possible), a leaked API key gives full agent access from any machine. The homedir field in the auth payload is client-supplied and could be spoofed — if the daemon uses it to resolve file paths, that's a directory traversal vector.

Section 07

Attack Surface

T2 LLM Inferred from RPC + schema + event correlation

We can classify the 82 RPC methods into three exposure tiers based on who can invoke them and what they can do:

User-facing methods (12) are the ones directly triggered by user actions in the CLI or GUI: add_user_message, request_permission, ask_user, interrupt_session, update_session_settings, session init/load, and the permission resolution chain. These are the primary interaction surface and the most tested code paths.

Daemon-internal methods (38) are the execution layer. Terminal operations, git operations, automation CRUD, session management, file system access. These are invoked by the daemon's own logic or by the agent's tool execution. The key risk here is that many of these methods perform irreversible operations: git_push, git_commit, create_pr, write_terminal_data, delete_automation. If the agent is compromised via prompt injection, these are the methods that cause real damage.

Read-only methods (18) are safe by design: listing sessions, terminals, files, MCP tools, automation history. They can leak information but can't modify state.

The MCP surface (14 methods) is the external extension interface and therefore the least trusted. MCP servers can provide arbitrary tools, and the toggle_mcp_tool method suggests granular enable/disable control — but once enabled, the tool inherits the agent's permissions.

⚠ What Could Go Wrong

The most dangerous combination is update_session_settings (which can set autonomyLevel to high) followed by daemon.write_terminal_data (which sends arbitrary commands to a PTY). If a prompt injection convinces the agent to first escalate its own permissions and then execute shell commands, the entire host is compromised. The permission system exists to prevent exactly this, but the proceed_auto_run_high level is a designed bypass.

The daemon.generate_semantic_diff + daemon.save_semantic_diff_cache pair is less obvious but still concerning: if the semantic diff cache is shared across sessions, a malicious diff summary could mislead a reviewer in a different session into approving a backdoored PR.

Section 08

Unknowns & Gaps

What the analysis couldn't determine

Every analysis has blind spots. Here's what we know we don't know, ranked by how much it would change our understanding if resolved:

?
Tunnel protocol details. We know tunnel.init and tunnel.ready exist, and that the computer CLI manages relay registrations. But we have zero visibility into the relay server, the authentication between endpoints, or the transport encryption. Is this a direct peer connection, or does it relay through a central server? The answer determines whether droid is a local-only tool or a distributed agent platform.
?
Rate limiting and quota enforcement. The pricing tier enum (team, payg, enterprise) tells us quotas exist, but we can't see the enforcement mechanism. Is it client-side (easily bypassed) or server-side? Does session_token_usage_changed report actual API token counts, or is it an abstraction?
?
Plugin system architecture. The plugin CLI subcommand exists but we found no plugin.* RPC namespace. Either plugins are managed entirely client-side, or the plugin API is hidden behind a different string pattern our extraction missed. Without this, we can't assess the plugin security model.
?
Session persistence format. The daemon manages sessions (archive_session, unarchive_session, search_sessions) but we don't know where they're stored. Local SQLite? Filesystem JSON? Cloud API? The search_sessions method suggests indexed search, which implies a database, but we can't confirm.
?
Automation scheduling backend. The schedule field in Gy0 is a cron expression, but we don't know if the daemon runs its own scheduler or delegates to the OS (systemd timers, launchd). The nextRunAt field suggests internal scheduling, but the isRetry field suggests retry logic which adds complexity.
?
The remaining 5 schemas. We detailed 4 of 9 Zod schemas (ax0, s6I, Gy0, by0). The other 5 are smaller and less distinctive, but they may contain MCP tool definitions, mission worker state, or git operation payloads. Further AST analysis could resolve these.
?
303 AST symbols. We focused on RPC methods, events, state machines, and schemas. The remaining ~150 symbols are a mix of internal utility functions, UI component names, and error strings. A systematic triage of these could reveal additional subsystems we haven't identified.
