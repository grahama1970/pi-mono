LLM Explanation
Architecture Role daemon.create_automation is the entry point for Droid's scheduled task system. It accepts a prompt (the LLM instruction), a schedule (cron expression), and a model (which LLM to use). Once created, the automation is stored server-side and executed by the daemon on schedule -- the client does not need to be connected.

Each automation run spawns a mission, which decomposes the prompt into features and assigns workers. This is the bridge between the simple "run this prompt at 3am" interface and the full mission orchestration engine. The get_automation_visual method suggests a timeline/DAG view exists in the GUI showing run history.

The 82-field Gy0 schema is the largest in the binary -- it covers CRUD, scheduling, execution tracking (durationMs, errorMessage, isRetry), and visual representation. The isRetry field indicates automatic retry logic on failure.

How does this relate to the mission system?
Each automation run creates a mission. The flow is:

create_automation (defines the task) → run_automation (triggers execution) → mission_accepted (mission engine takes over) → workers decompose the prompt into features.

The automationId in Gy0 links to the mission's sessionId, so you can trace any mission back to the automation that spawned it.
