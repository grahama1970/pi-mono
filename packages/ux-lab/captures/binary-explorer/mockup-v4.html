What is this? LLM
Creates a new scheduled automation — a cron-triggered agent task that runs without human input. The caller provides a prompt (the instruction), a schedule (cron expression), and a model (which LLM to use). The daemon validates, assigns an automationId, and schedules it.

This is the entry point for droid's "set it and forget it" mode — recurring tasks like "review PRs every morning" or "run tests on push."

What can automations do?
Automations are cron-scheduled agent sessions. You give them a prompt, a model, and a schedule — the daemon runs them unattended.

The CRUD surface is complete: create, run, pause, resume, rename, update_model, delete. Each run is tracked with startedAt, completedAt, durationMs, and errorMessage.

Failed runs auto-retry (isRetry + originalRunId fields). There's also a visual timeline (get_automation_visual).
Evidence: schema Gy0 (82 fields) + 7 daemon.* methods
Can they trigger missions?
Yes — automations and missions share the same execution path. When daemon.run_automation fires, it emits mission_accepted → mission_run_started, then fans out to workers. State machine M tracks the lifecycle.

The connection: automationId appears in both the Gy0 automation schema AND the mission event payloads. An automation IS a mission with a cron schedule attached.
Evidence: shared automationId field + mission_* events
Is this exposed without auth?
No. daemon.authenticate is required before any daemon method. Schema ax0 shows the auth handshake: token or apiKey + userId/orgId. The connectionId field suggests per-connection auth state.

All automation methods are in the daemon.* namespace — not droid.* — meaning they require the daemon connection, which requires auth.
